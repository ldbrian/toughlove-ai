ToughLove v2.3.0 战略规划文档：命运与剧本
版本代号： v2.3.0 "Fate & Narrative" (命运与叙事) 战略核心： 从“被动聊天工具”转型为“主动内容提供者” 核心哲学： AI 是导演，用户是演员；用“剧本”消灭“输入框恐惧症”。

1. 战略定位转型 (Strategic Pivot)
1.1 痛点诊断
输入门槛过高： 纯聊天模式要求用户充当“内容生产者”，认知负荷过大，导致冷启动流失。

缺乏回访理由： 如果没有新内容，用户就没有每天打开 App 的动力。

1.2 新定位：诱导式交互 (Guided Interaction)
内容前置： App 主动向用户推送“今日剧本”（塔罗+运势+任务），而非等待用户开口。

辅助聊天： 聊天功能退居二线，作为内容的延伸和情绪的宣泄口。

命运感： 通过随机事件与五维人格的组合，营造“冥冥之中”的宿命感，增强用户粘性。

2. 核心功能：每日晨报 (Daily Briefing)
取代原有的“每日毒签”和“主动日记”，成为 App 的第一入口。

2.1 业务流程 (User Flow)
自动触发 (Encounter)： 用户每日首次打开 App，Echo 自动接管屏幕（覆盖首页），无需点击。

闭环反馈 (Reality Check)：

Echo 询问：“昨天的剧本（塔罗牌）准吗？”

用户操作：点击“准”或“不准”。

价值：收集用户真实生活数据，修正后台算法。

内容分发 (The Script)：

Echo 抽取“今日塔罗牌”。

聚光灯机制 (Spotlight)： 系统根据羁绊值（Trust Level）或随机算法，选出一位**“今日主讲人”**（Ash/Rin/Sol）。

人格反应： 主讲人针对这张牌发表专属评论（毒舌/关怀/任务）。

无缝入戏 (The Jump)：

用户点击行动按钮：“去找 [主讲人] 聊聊”。

系统自动跳转至该人格的聊天窗口。

记忆注入 (Context Injection)： 后台自动插入一条系统指令，让 AI 知道刚才发生了什么，并顺着话题开聊。

2.2 界面与交互 (UI/UX)
视觉风格： 神秘、沉浸、甚至带有一点压迫感。

操作成本： 极低。用户只需点击选项或按钮，无需打字即可完成整个晨会流程。

3. 数据与内容架构 (Content Architecture)
3.1 塔罗牌数据结构升级
在 src/lib/constants.ts 中，TAROT_DECK 需扩展为包含多维度反应的结构。

TypeScript

interface TarotCard {
  id: number;
  name: { zh: string; en: string };
  keyword: string;
  meaning: string;
  reactions: {
    Ash: string; // 毒舌辣评
    Rin: string; // 傲娇/陪伴/小任务
    Sol: string; // 秩序/计划/警告
    Vee: string; // 嘲讽/混乱/梗
    Echo: string; // 深度/本质
  };
}

4. 未来展望：五维接力 (The Relay) - v2.4 预研
这是为了解决“长对话枯燥”问题的下一步规划。

概念： 情绪过山车。

机制： 当一段对话结束（或用户流失）时，系统自动生成“交接棒”。

场景示例：

用户被 Ash 骂完（情绪低落/压力大）。

Ash 隐退，Sol 接棒：“既然被骂醒了，那就来做个任务证明自己。”

任务完成，Rin 接棒：“做得好！给你奖励。”

目的： 利用五个人格的不同属性，覆盖用户一整天的情绪周期，形成“不得不回访”的闭环。
